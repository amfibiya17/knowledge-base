# 9.2 Commit Message Conventions

This file describes a simple and consistent convention for writing Git commit messages.

---

## feat

Use when adding **new functionality** or behaviour.

```bash
git commit -m "feat: add login form"
git commit -m "feat: implement search filtering"
```

## fix

Use when fixing a **real bug** (something was broken and now works).

```bash
git commit -m "fix: handle empty search results"
git commit -m "fix: prevent crash when user is undefined"
```

## docs

Use for **documentation-only** changes (README, notes, comments).

```bash
git commit -m "docs: add virtual dom explanation"
git commit -m "docs: update git workflows notes"
```

## chore

Use for **maintenance tasks** that donâ€™t change behaviour  
(e.g. linting, formatting, tooling, config).

```bash
git commit -m "chore: run linter"
git commit -m "chore: update dependencies"
```

## refactor

Use when **restructuring code** without changing behaviour  
(improving readability, structure, or design).

```bash
git commit -m "refactor: simplify conditional logic"
git commit -m "refactor: extract reusable helper function"
```

## test

Use when **adding or updating tests**.

```bash
git commit -m "test: add unit tests for login form"
git commit -m "test: improve coverage for utils"
```
